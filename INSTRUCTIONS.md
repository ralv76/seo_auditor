# Инструкция: SEO-аудит со сниженным расходом токенов

## Принцип

Разделить процесс на два этапа:
1. **Сбор сырых данных** — выполняется скриптами Python (Playwright + BeautifulSoup / requests). Токены LLM не расходуются.
2. **Анализ и формирование отчета** — выполняется LLM по агрегированным данным. Расход токенов снижен, так как на вход подается не HTML, а структурированный JSON.

## Этап 1. Сбор данных (Python, 0 токенов)

### 1.1. Crawler — обход сайта

**Инструмент:** Playwright (sync_api) или requests + BeautifulSoup.
**Что собираем:**
- HTML каждой страницы
- HTTP-заголовки (content-type, x-robots-tag)
- Статус ответа (200, 301, 404, 5xx)
- robots.txt, sitemap.xml

**Код:** `crawler.py` (фреймворк ниже).

**Выход:** `raw_data.json` — массив объектов по каждому URL.

```json
{
  "url": "https://4info.ru/",
  "status": 200,
  "html": "<!DOCTYPE html>...",
  "headers": {"content-type": "text/html"},
  "title": "...",
  "meta": {"description": "...", "keywords": "..."},
  "headings": {"h1": [...], "h2": [...]},
  "links": {"internal": [...], "external": [...]},
  "images": [{"src": "...", "alt": "..."}],
  "schemas": [...],
  "canonical": "...",
  "robots_meta": "...",
  "analytics_scripts": ["yandex", "google"],
  "word_count": 377
}
```

### 1.2. Extractor — парсинг HTML

**Инструмент:** BeautifulSoup (lxml).
**Что извлекаем:**
- title, meta description, keywords, viewport, canonical
- Open Graph (og:title, og:description, og:image, og:type)
- Twitter Cards (twitter:card, twitter:title и др.)
- Заголовки H1-H6
- Ссылки (внутренние / внешние, nofollow)
- Изображения (src, alt, width, height)
- Schema.org (JSON-LD)
- Analytics (наличие скриптов по сигнатурам)
- Количество слов (текст без тегов)

**Код:** встроен в `crawler.py` как функции `extract_*`.

### 1.3. Performance metrics

**Инструмент:** Playwright + Performance API.
**Что измеряем:**
- TTFB (responseStart - startTime)
- FCP (first-contentful-paint)
- LCP (largest-contentful-paint)
- CLS (cumulative layout-shift)
- DOMContentLoaded
- LoadEventEnd
- TransferSize, EncodedBodySize

**Код:** JavaScript-инъекция через Playwright `page.evaluate`.

### 1.4. robots.txt и sitemap.xml

**Инструмент:** requests.
**Что проверяем:**
- Наличие файлов
- Валидность синтаксиса robots.txt
- Количество URL в sitemap
- Наличие изображений в sitemap (рекомендация: вынести)

### 1.5. Поиск конкурентов (опционально)

**Вариант A (автоматический):**
- Python-скрипт ищет в поисковой выдаче (Yandex/Google) по ключевым словам, извлеченным из meta keywords главной страницы.
- Парсит выдачу через Playwright или SERP API.
- Собирает топ-5 доменов.

**Вариант B (ручной):**
- Пользователь задает список конкурентов в `competitors.txt`.
- Скрипт только собирает их данные аналогично этапу 1.1.

**Токены LLM:** не требуются.

## Этап 2. Агрегация (Python, 0 токенов)

**Инструмент:** pandas / стандартный Python.
**Что считаем:**
- Количество страниц с уникальным title / description
- Количество страниц без H1 / с дублями H1
- Средний объем текста
- Процент изображений без alt
- Количество внутренних / внешних ссылок
- Средние значения Core Web Vitals
- Сравнительная таблица с конкурентами

**Выход:** `aggregated.json` — сводные метрики.

```json
{
  "pages_total": 18,
  "pages_without_description": 1,
  "pages_with_duplicate_h1": 2,
  "avg_word_count": 1200,
  "images_without_alt_percent": 5.2,
  "core_web_vitals": {"ttfb_ms": 7, "fcp_ms": 384, "cls": 0},
  "competitors": {
    "tilda": {"avg_word_count": 464, "images_without_alt_percent": 100},
    "tochka": {"avg_word_count": 1690, "images_without_alt_percent": 48}
  }
}
```

## Этап 3. Формирование отчета (LLM, минимальные токены)

**Вход:** `aggregated.json` + несколько примеров сырых данных (не весь HTML).
**Выход:** `SEO_AUDIT.md`.

### Что делает LLM:
- Интерпретирует цифры ("TTFB 7 мс — отлично").
- Сравнивает с конкурентами.
- Формулирует рекомендации на естественном языке.
- Приоритизирует проблемы.

### Что НЕ делает LLM:
- Не парсит HTML.
- Не считает метрики.
- Не обходит страницы.

### Оптимизация промпта:

```markdown
Ты — SEO-аналитик. Ниже агрегированные данные аудита сайта {domain}.
Сформируй отчет в Markdown со следующими разделами:
1. Резюме (таблица ключевых параметров)
2. Технический SEO (мета-теги, заголовки, схемы)
3. Контент (объем, ключевые слова, изображения)
4. Производительность (Core Web Vitals)
5. Анализ конкурентов (сравнительная таблица)
6. Приоритетные рекомендации (критические, важные, опциональные)

Данные:
{aggregated_json}

Требования:
- Используй таблицы где возможно
- Не повторяй очевидное
- Каждая рекомендация должна содержать конкретный URL или тип страницы
```

## Распределение задач по исполнителям

| Задача | Python | LLM | Почему |
|--------|--------|-----|--------|
| Обход страниц | **Да** | Нет | Механическая работа |
| Парсинг HTML | **Да** | Нет | Регулярки / DOM |
| Подсчет метрик | **Да** | Нет | Арифметика |
| Сравнение с конкурентами | **Да** | Нет | Табличные данные |
| Интерпретация результатов | Нет | **Да** | Требуется экспертиза |
| Формулировка рекомендаций | Нет | **Да** | Требуется языковая модель |
| Приоритизация проблем | Нет | **Да** | Требуется контекст |
| Написание отчета | Нет | **Да** | Генерация текста |

## Оценка экономии токенов

| Подход | Примерный расход токенов | Время |
|--------|--------------------------|-------|
| Только LLM (как в текущем аудите) | ~50K-100K токенов (HTML каждой страницы передается в контекст) | 15-20 мин |
| Python + LLM (по инструкции) | ~2K-5K токенов (только JSON) | 3-5 мин на сбор + 1 мин на анализ |
| Экономия | **90-95%** | **В 3-4 раза быстрее** |

## Ключевое правило

**HTML и сырые тексты не передаются в LLM.**
Передаются только:
- Числовые метрики
- Списки (URL, заголовки, ключевые слова)
- Булевы флаги (есть/нет)
- Сравнительные таблицы

Таким образом, LLM работает как аналитик, а не как краулер.
